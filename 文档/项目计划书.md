# 🏋️‍♂️ Smart-Fit 智能健身辅助系统 - 项目计划书

## 一、 项目概述
本项目旨在开发一款基于大语言模型（LLM）的智能健身辅助**微信小程序**。针对传统健身软件“计划死板、缺乏个性化指导”的痛点，通过 AI 动态生成专属训练计划与 24 小时智能答疑，并结合“本地高质量动作库”保障指导的安全专业性，打造低门槛的智能健身管家。

## 五、 开发模式与工具选型

### 5.1 开发模式：敏捷开发 
结合 10 周的开发周期，本项目采用**敏捷开发模式**，将整个周期划分为 5 个 Sprint（迭代冲刺），每个 Sprint 为期 2 周。
*   **每周站会 **：每周三在课上做简报：“上周做了什么，遇到了什么困难，本周计划做什么，”。
*   **代码审查 **：合并代码到主分支前，前后端内部互相 Review，保证代码质量。

### 5.2 工具选择
*   **项目协作与任务看板**：**飞书 (Lark)** 。
*   **产品原型设计**：**墨刀** 或 **Figma**。
*   **版本控制与代码托管**：**Git + Gitee (码云)**。
*   **API 接口管理与调试**：**Apifox** 或 **Postman**。
*   **开发工具 (IDE)**：
    *   前端：微信开发者工具 + VS Code。
    *   后端：PyCharm / IntelliJ IDEA。

### 5.3 平台选择 
*   **目标运行平台**：**微信小程序**（基于微信开发者工具进行“真机调试”，跨平台且无需繁琐的安装过程，演示效果最佳）。
*   **后端开发语言**：**Python (FastAPI / Flask)**.
*   **数据库平台**：**MySQL 8.0**。
*   **AI 能力平台**：**阿里云百炼平台（通义千问）** 或 **智谱 AI 开放平台（GLM-4）**。

## 六、 风险应对 
### 6.1 技术风险 

| 风险描述 | 可能性/影响 | 应对措施 | 负责人 |
| :--- | :---: | :--- | :--- |
| **AI 动作幻觉风险**：AI 生成了本地库中不存在的动作名称（如“单手倒立俯卧撑”）。 | 高 / 高 | 1. **Prompt 约束**：在 System Prompt 中喂入本地动作 ID 清单，强制 AI 仅从中选择。<br>2. **后端模糊匹配算法**：若 AI 返回动作不在库中，使用 Levenshtein 距离算法匹配名称最接近的本地动作。<br>3. **兜底方案**：若完全无法匹配，系统自动推送一套预设的“经典全身唤醒”计划，防止界面报错。 | BE2, PM |
| **JSON 解析失败**：AI 返回格式不符合约定（如混杂了多余的文本），导致系统崩溃。 | 中 / 高 | 1. **强制格式化**：使用大模型特定的 `json_mode` 或在 Prompt 结尾加 `Response in strict JSON format`。<br>2. **健壮性校验**：后端增加 JSON Schema 验证层，解析失败则自动重试一次或调用备用模型。 | BE2 |
| **AI 响应延迟过高**：生成一份复杂计划需 10-20秒，用户流失严重。 | 高 / 中 | 1. **流式传输 (Stream)**：对话界面采用流式输出，让用户即时看到文字跳动。<br>2. **感知优化**：在生成计划时，前端展示“AI 教练正在排课中...”的健身小人 Loading 动画。<br>3. **异步处理**：用户提交请求后可离开，生成完毕后通过微信模板消息通知（若有权限）或在首页红点提醒。 | FE2, BE2 |

### 6.2 业务与安全风险 

| 风险描述 | 可能性/影响 | 应对措施 | 负责人 |
| :--- | :---: | :--- | :--- |
| **健身安全性风险**：AI 给有伤病（如腰突）的用户推荐了危险动作（如硬拉）。 | 低 / 极高 | 1. **预设硬性过滤**：在档案中加入“伤病史”字段。后端逻辑在调用 AI 前，通过代码层面过滤掉高危动作 ID。<br>2. **安全免责声明**：在生成计划前强制用户阅读并勾选安全提示。<br>3. **难度分级**：Prompt 根据用户运动等级（初/中/高）严格限制动作库检索范围。 | PM, BE1 |
| **动作姿势指导错误**：AI 对动作要领的解释与标准动图不符。 | 低 / 中 | 1. **静态关联**：前端渲染时，文字描述优先读取本地库中专家审核过的“动作说明”，而非由 AI 随机生成。<br>2. **素材审核**：由测试工程师 (QA) 逐一核对动作 ID 与 Gif 图、教学文字的一致性。 | QA, FE2 |
| **用户隐私数据泄露**：用户的身高、体重、照片等敏感数据外泄。 | 低 / 高 | 1. **鉴权机制**：所有接口必须携带 JWT Token 校验，防止越权访问他人档案。<br>2. **数据脱敏**：数据库中的敏感信息进行加密存储。 | BE1 |

| 风险描述 | 可能性/影响 | 应对措施 | 负责人 |

### 6.4 资源风险 

| 风险描述 | 可能性/影响 | 应对措施 | 负责人 |
| :--- | :---: | :--- | :--- |
| **API 调用费用超支/欠费停服** | 中 / 中 | 1. **本地缓存策略**：对于相同画像（如：男/25岁/增肌）的初始请求，后端缓存首份计划，减少 API 重复调用。<br>2. **限流保护**：设置单个用户每日调用 AI 的次数上限（如 5 次）。<br>3. **多模型备选**：同时接入 DeepSeek 和 通义千问。若一家 API 异常或超载，自动切换备用。 | BE2 |
| **云服务器宕机/数据库故障** | 低 / 中 | 1. **自动化备份**：编写脚本，每周自动备份一次数据库 SQL 文件。<br>2. **监控报警**：使用云平台自带的监控告警，CPU 占用过高即时通知负责人。 | BE1 |
