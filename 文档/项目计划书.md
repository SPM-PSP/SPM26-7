# 🏋️‍♂️ AI-Fit 智能健身辅助系统 - 项目计划书

## 一、 项目概述
本项目旨在开发一款基于大语言模型（LLM）的智能健身辅助**微信小程序**。针对传统健身软件“计划死板、缺乏个性化指导”的痛点，通过 AI 动态生成专属训练计划与 24 小时智能答疑，并结合“本地高质量动作库”保障指导的安全专业性，打造低门槛的智能健身管家。

---

## 二、 团队组织架构与职责划分 (5人团队)

| 角色 | 成员 | 核心职责说明 |

## 三、 开发模式与工具选型

### 1. 开发模式：敏捷开发 (Agile - Scrum)
结合 8 周的开发周期，本项目采用**敏捷开发模式**，将整个周期划分为 4 个 Sprint（迭代冲刺），每个 Sprint 为期 2 周。
*   **每周站会 (Weekly Stand-up)**：每周三在课上做简报：“上周做了什么，遇到了什么困难，本周计划做什么，”。
*   **代码审查 (Code Review)**：合并代码到主分支前，前后端内部互相 Review，保证代码质量。

### 2. 工具选择 (Tools Stack)
*   **项目协作与任务看板**：**飞书 (Lark)** 或 **Teambition**（用于分配需求、跟踪 Bug）。
*   **产品原型设计**：**墨刀** 或 **Figma**（PM 用于画界面草图）。
*   **版本控制与代码托管**：**Git + Gitee (码云)**。
*   **API 接口管理与调试**：**Apifox** 或 **Postman**（强烈推荐 Apifox，后端写完接口直接生成文档，前端可以直接 Mock 数据测试）。
*   **开发工具 (IDE)**：
    *   前端：微信开发者工具 + VS Code。
    *   后端：PyCharm / IntelliJ IDEA。

### 3. 平台选择 (Platform Selection)
*   **目标运行平台**：**微信小程序**（基于微信开发者工具进行“真机调试”，跨平台且无需繁琐的安装过程，演示效果最佳）。
*   **后端开发语言**：**Python (FastAPI / Flask)** 或 **Java (Spring Boot)**。
*   **数据库平台**：**MySQL 8.0**。
*   **AI 能力平台**：**阿里云百炼平台（通义千问）** 或 **智谱 AI 开放平台（GLM-4）**。

## 四、 项目预算估算 

本项目主打低成本落地，主要开销在于后端的云服务器部署和大模型的 API 调用，整体预算可控制在 **100元以内**。

| 支出项 | 方案说明 | 预估金额 |
| :--- | :--- | :--- |
| **云服务器 (部署后端)** | 腾讯云/阿里云“轻量应用服务器”（学生机或新用户特惠），购买 1-2 个月即可满足开发和答辩需求。 | ~ 30-60 元 |
| **大模型 API 费用** | 智谱 GLM-4 / 阿里通义千问 / DeepSeek。各大平台对新注册开发者通常赠送 100万-500万免费 Token，测试开发阶段基本**白嫖**。即使超出，按量计费也非常便宜。 | ~ 10-20 元 |
| **微信小程序认证** | 仅使用“测试号”或“体验版”进行答辩演示，无需正式上线发布。 | 0 元 |
| **域名与备案** | 直接通过云服务器的 IP 地址进行接口调用（配置小程序局域网/开发模式白名单即可），无需购买域名。 | 0 元 |
| **总计** | | **约 40 - 80 元** |

## 五、 8 周项目进度里程碑 (8-Week Schedule)

### 🟢 第 1 - 2 周：Sprint 1（需求定稿与基础设施铺建）
*   **PM**：完成《需求规格说明书》；使用墨刀画出所有界面的原型图；在 Gitee 创建代码仓库；申请 AI 平台 API Key。
*   **前端组**：搭建小程序项目骨架；配置 Vant UI 组件库；完成底部的全局导航栏。
*   **后端组**：敲定数据库实体关系图 (ER图)；在云服务器上装好 MySQL；搭建后端基础框架（跑通 Hello World）；在 Apifox 建立接口文档。

### 🟡 第 3 - 4 周：Sprint 2（MVP 基础功能开发）
*   *目标：不带 AI 的功能必须全部跑通。*
*   **前端组**：完成“用户注册/登录”页面、“个人档案”表单页、“静态动作库”列表页、“日历打卡”UI 开发。
*   **后端组**：完成基础业务的 CRUD 操作（用户的增删改查、动作图片的返回）；收集并录入本地动作库数据。
*   **PM**：编写并在本地测试 AI 提示词（Prompt），测试能否稳定生成 JSON 格式的计划。

### 🟠 第 5 - 6 周：Sprint 3（AI 核心功能攻坚 - 难度最高）
*   *目标：前后端全面联调，AI 功能落地。*
*   **后端组**：全面接入大模型 API；实现“AI 计划生成接口”（难点：处理 AI 返回格式错误的异常）；实现“AI 意图识别+本地动作库匹配”接口。
*   **前端组**：开发动态“计划任务卡片”渲染逻辑；开发“24小时 AI 教练”的聊天气泡交互界面（难点：处理流式输出打字机效果或 Loading 动画）。
*   **PM**：监控前后端接口对接进度，开始第一轮系统级粗测。

### 🔴 第 7 周：Sprint 4（UI 优化与系统测试）
*   *目标：系统缝合，美化界面，排除致命 Bug。*
*   **全员参与**：代码冻结（Code Freeze），不再开发新功能。
*   **前端组**：修复页面在不同型号手机上的 UI 错位问题；优化加载动画。
*   **后端组**：优化数据库查询速度；处理前端提测的 Bug（如重复点击打卡报错等）。
*   **PM**：进行边界测试（输入异常的身高体重、在 AI 聊天里输入奇怪的问题），确保系统不会崩溃。

### 🟣 第 8 周：交付与答辩演练（终极冲刺）
*   **系统部署**：后端代码正式稳定运行在云服务器上。
*   **视频录制 (极其重要)**：录制一段完美运行的 Demo 演示视频，配上字幕，防止答辩现场由于网络差导致大模型接口调用失败。
*   **文档整理**：PM 整理汇总《项目总结报告》、API 接口文档、数据库字典。
*   **答辩准备**：制作精美的 PPT（突出重点：你们是如何利用“意图识别+本地库”规避 AI 画图幻觉的）。模拟答辩排练。

## 六、 质量把控与风险管理
1. **接口对接延期风险**：采用 Apifox 进行 Mock 开发。后端即便没写完逻辑，前端也能根据约定好的假数据（Mock Data）进行 UI 渲染，互不阻塞。
2. **AI 不可控风险**：后端必须对大模型返回的数据加上 `try-catch` 解析。一旦大模型抽风返回了非 JSON 数据，系统立刻启用“兜底预案”（返回一套写死在代码里的默认健身计划），绝不让用户在前端看到乱码。
