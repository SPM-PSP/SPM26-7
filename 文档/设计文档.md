# 1. 概述

## 1.1 项目背景
菲特勒斯是一个综合性的健康管理和AI辅助系统，旨在通过先进的人工智能技术为用户提供个性化的健康建议、营养指导、运动计划和心理健康支持。系统整合了食物记录、运动追踪、健康数据分析、AI聊天机器人等多种功能，为用户提供全方位的健康管理解决方案。

## 1.2 系统定位
- **健康管理平台**：提供全面的健康数据记录和分析
- **AI辅助系统**：通过AI技术提供个性化建议和指导
- **用户交互中心**：建立用户与AI之间的智能对话系统
- **数据驱动平台**：基于用户数据提供精准的健康建议

## 1.3 核心价值
- **个性化**：根据用户历史数据提供定制化建议
- **智能化**：利用AI技术优化健康管理流程
- **全面性**：覆盖营养、运动、心理等多个健康领域
- **易用性**：简洁直观的用户界面和交互体验

## 2.6 数据库设计
### 2.6.1 数据库选型
- **关系型数据库**：SQLITE - 用于结构化数据存储
- **部署模式**：主从复制，读写分离
- **数据存储策略**：
  - 用户数据：持久化存储在SQLITE中
  - AI记忆：使用IndexedDB进行客户端本地存储
  - 缓存数据：Redis用于高速缓存 （后期扩张工资）
- **数据库连接**：通过API路由与数据库交互


### 2.6.2 核心表结构
- `user_profiles` 用户表
- `weight_records` 体重记录表
- `activity_level_records` 活动水平表

- `exercise_entries` 运动记录表

### 2.6.4 数据关系设计
- **用户表（user_profiles）** 与 **体重记录表（weight_records）**、**活动水平记录表（activity_level_records）**、**食物记录表（food_entries）**、**运动记录表（exercise_entries）**、**专家对话表（chat_{expert_role}）**、**建议历史表（suggestion_history）** 之间存在一对多的关系，通过 user_id 字段进行关联。
- **专家对话表（chat_{expert_role}）** 之间的表结构相同，只是表名不同，分别对应不同的专家角色。

# 5 性能指标设计
## 5.1 响应时间指标
1. 通用接口请求整体响应时间 ≤ 2s；健康日志提交、本地数据同步操作响应 ≤ 1s。
2. 卡路里计算、AI健康建议生成、长期数据趋势分析报告加载响应时间 ≤ 3s。
3. 对话交互、记忆读取与上下文加载响应时间 ≤ 1.5s，保障多轮对话流畅性。

## 5.2 数据处理能力指标
1. 单用户单日支持50条及以上饮食、运动、状态日志记录的存储、解析与检索。
2. 累计1年历史健康数据整合分析、可视化图表渲染耗时 ≤ 5s。
3. 适配学生项目使用场景，支持同时在线用户数≥200，系统无明显卡顿、页面无阻塞。
4. 内置食物数据库、用户记忆数据轻量化加载，减少重复计算资源消耗。

## 5.3 图片处理性能指标
1. 食物拍摄图片压缩、解析、AI识别全流程处理时间 ≤ 5s，支持1080P高清图片无损压缩。
2. 数据分析截图、报告导出截图生成耗时 ≤ 2s，满足日常截图留存需求。
3. 支持单张图片轻量化预处理，降低前端与后端运算压力。

## 5.4 系统稳定与资源指标
1. 系统持续运行稳定性良好，无频繁崩溃、卡死、内存溢出问题。
2. 优化内存调度机制，规避内存泄漏问题，长时间后台运行资源占用可控。
3. 本地缓存、数据读写高效稳定，保障离线基础功能正常使用。

# 6 硬件设计
## 6.1 客户端硬件要求（学生通用设备）
### 6.1.1 移动端设备
- 最低配置：Android 8.0 / iOS 12.0 及以上系统，运行内存2GB，剩余存储空间≥200MB，后置摄像头800万像素及以上。
- 推荐配置：Android 10.0 / iOS 14.0 及以上系统，运行内存4GB，剩余存储空间≥500MB，满足图片识别与动画渲染需求。

### 6.1.2 桌面端设备
- 最低配置：双核处理器，内存4GB，硬盘剩余空间100MB以上，支持主流浏览器正常运行。
- 推荐配置：四核处理器，内存8GB，固态硬盘存储，保证数据可视化、多页面切换流畅运行。

## 6.2 服务端硬件要求（学生项目轻量化部署）
1. 核心服务设备：采用普通个人计算机/低配云服务器，双核CPU、8GB内存、500GB机械硬盘/256GB固态硬盘，满足项目基础运行。
2. 数据存储：本地文件+轻量数据库存储，无需专业磁盘阵列，满足小批量学生用户数据持久化。
3. AI调用与运算：依托第三方AI在线API，本地设备仅做数据接收与解析，无需独立GPU算力硬件。
4. 网络环境：普通校园网、家庭宽带即可满足接口请求、图片上传、数据同步网络带宽需求。

## 6.3 辅助硬件要求
1. 无额外专用硬件依赖，完全适配日常学习、生活通用电子设备。
2. 依赖终端设备摄像头完成食物拍照识别，依赖网络完成AI接口通信。

# 7 其他设计
## 7.1 系统质量属性设计
### 7.1.1 安全性
1. 遵循数据加密规范，用户健康隐私数据、配置信息采用加密方式本地存储，传输过程使用HTTPS协议。
2. 完善输入内容校验与过滤机制，防范非法输入、脚本注入等基础安全风险。
3. 设计基础权限访问控制，用户仅可查看、编辑个人健康数据与AI记忆内容，隔离多用户数据。
4. 敏感健康信息、自定义档案内容单独加密存储，保障个人隐私合规性。

### 7.1.2 可靠性
1. 采用本地持久化存储结合云端临时同步方案，依托IndexedDB实现本地数据留存，防止数据意外丢失。
2. 完善全局异常捕获与错误处理机制，操作异常时展示友好提示，避免程序直接崩溃。
3. 搭建简易日志记录机制，记录操作异常、接口请求错误，便于项目调试与问题修复。
4. 支持基础数据手动备份、导出恢复功能，提升数据使用可靠性。

### 7.1.3 易用性
1. 全平台响应式布局设计，适配手机、平板、电脑多类终端设备，界面自适应缩放。
2. 支持中英文双语切换、深浅色主题切换，满足不同使用习惯与场景需求。
3. 界面交互动画流畅，保证60fps基础帧率，简化操作流程，降低新手使用学习成本。
4. 功能布局简洁清晰，搭配操作提示、文字说明，提升整体使用体验。

### 7.1.4 可维护性
1. 整体采用模块化、分层化开发设计，功能解耦，便于后期功能迭代、修改与扩展。
2. 统一接口规范与代码注释，标准化命名规则，适配学生项目开发维护需求。
3. 功能模块独立拆分，健康记录、数据分析、AI对话、记忆系统互不耦合，便于单独调试。

## 7.2 AI Memory系统设计
### 7.2.1 整体架构设计
采用多专家独立记忆架构，划分营养师、健身教练、健康顾问独立记忆空间，同时搭建公共共享记忆层与上下文管理层，实现记忆隔离、同步与调度，规避不同角色信息混淆干扰。

### 7.2.2 记忆管理设计
1. 记忆持久化：所有AI角色记忆数据本地持久化存储，支持长期保存，重启系统不丢失数据。
2. 记忆版本与内容管理：限制记忆文本长度与格式规范，轻量化存储，同时保留记忆变更基础记录，实现简易版本回溯。
3. 独立记忆空间：为不同AI专家角色分配隔离记忆存储空间，保证各角色专业建议的独立性。
4. 记忆共享与清理：支持多角色间基础用户数据共享，实现综合健康分析；提供手动清理指定角色记忆、批量清空记忆功能，兼顾隐私管理。

### 7.2.3 上下文管理设计
1. 标准化上下文协议适配，维持对话上下文连续性，实现跨会话长期记忆留存。
2. 建立记忆实时更新机制，用户健康数据、对话内容变更后，自动同步更新至AI记忆库。
3. 优化轻量化检索机制，通过用户标识、角色标识快速检索对应记忆内容，保障对话响应效率。
4. 支持多角色对话上下文快速切换，无缝切换营养师、健身顾问对话场景。

### 7.2.4 接口与安全设计
1. 提供标准化记忆增删改查接口，支撑记忆读取、更新、同步、备份全流程操作。
2. 增加记忆数据格式校验、内容验证机制，过滤无效数据，保证记忆内容规范有效。
3. 完善记忆访问权限控制，仅授权AI角色可读取对应记忆内容，敏感记忆加密存储。
4. 配套记忆数据备份与恢复功能，支持手动导出记忆数据，防止个性化记忆内容丢失。

### 7.2.5 联动交互设计
1. 智能对话系统与AI记忆深度联动，多轮对话内容自动同步入库，持续完善用户健康画像。
2. AI依托历史记忆、7天健康数据、多维度生活状态信息，生成贴合用户实际的个性化建议。
3. 依托动态记忆能力，记录用户饮食偏好、运动习惯、健康目标，持续优化AI建议精准度。
