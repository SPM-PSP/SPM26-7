# 1. 概述

## 1.1 项目背景
菲特勒斯是一个综合性的健康管理和AI辅助系统，旨在通过先进的人工智能技术为用户提供个性化的健康建议、营养指导、运动计划和心理健康支持。系统整合了食物记录、运动追踪、健康数据分析、AI聊天机器人等多种功能，为用户提供全方位的健康管理解决方案。

## 1.2 系统定位
- **健康管理平台**：提供全面的健康数据记录和分析
- **AI辅助系统**：通过AI技术提供个性化建议和指导
- **用户交互中心**：建立用户与AI之间的智能对话系统
- **数据驱动平台**：基于用户数据提供精准的健康建议

## 1.3 核心价值
- **个性化**：根据用户历史数据提供定制化建议
- **智能化**：利用AI技术优化健康管理流程
- **全面性**：覆盖营养、运动、心理等多个健康领域
- **易用性**：简洁直观的用户界面和交互体验

## 2.2 技术栈

1. 核心框架
全栈框架：Next.js 15.2.4 (基于 App Router 架构，统一前后端路由与渲染)
开发语言：TypeScript + Node.js (提供全链路的类型安全)
2. 前端 
样式解决方案：Tailwind CS
UI 基础组件库：Radix UI (无头组件库 Headless UI，提供极佳的无障碍访问和自定义能力)
国际化方案：next-intl 
3. 逻辑与状态管理 
全局状态管理：React Context API (轻量级，替代 Redux 处理全局状态)
表单处理：React Hook Form (高性能表单状态管理与校验)
4. 数据与通信层
数据库：SQLite (轻量级本地文件数据库，极大地降低了部署和运维成本)
接口规范：HTTP / RESTful API (配合 Next.js 的 Route Handlers)
数据交换格式：JSON

5. 工程化
构建工具：Next.js 内置 Webpack/Turbopack 构建系统

部署架构：全栈一体化部署 (单体应用模式，极大概率会托管在 Vercel 或通过 Docker 容器化部署)
开发环境：VS Code

### 2.6.4 数据关系设计
* **核心挂载点**：所有业务流水表（健康时序指标表、食物记录表、运动记录表、大模型对话表、长期记忆库、建议历史表）均通过 `user_id` 与 **用户核心档案表（user_profiles）** 形成一对多的强关联约束。
* **上下文闭环**：**统一大模型对话表（ai_chat_messages）** 内部通过 `session_id` 串联出完整的单次多轮对话上下文；同时，**AI 长期记忆特征库（ai_long_term_memories）** 通过 `source_message_id` 字段反向关联至对话表，实现 AI 记忆生成的可解释性与溯源。
* **非结构化扩展**：利用 SQLite 对 JSON 的原生支持，系统可在不频繁执行数据库表结构变更（Migration）的情况下，动态向各业务表的 JSON 字段中追加新的标签与指标数据，提升后续功能的迭代速度。

## 2.7 系统交互设计
##### 数据流设计
系统的数据流转严格遵循单向数据流（Unidirectional Data Flow）与读写分离原则。考虑到系统包含高频的 UI 交互和耗时的 AI 推理，数据流设计按业务场景拆分为三种核心流转模式：

**前端状态数据流 (Client State Flow)**
在 Next.js 的客户端组件（Client Components）中，严格遵循 React 的单向数据流闭环：
* **Action 触发**：用户操作（如点击拍照上传食物或提交当前体重）触发前端事件处理器。
* **数据上报**：前端通过 Next.js Server Actions 或 Fetch API 将 Payload 发送至应用网关层，在此层完成 JWT 鉴权与 API 防刷限流。
* **State 更新**：收到后端领域业务层的成功响应后，将最新健康数据派发（Dispatch）给 React Context 全局状态。
* **UI 重绘**：状态变更驱动 Radix UI 组件库重新渲染（如更新健康看板），确保界面的数据源唯一性（Single Source of Truth）。

**AI 专家多轮对话与推理流 (AI RAG & Streaming Flow)**
由于大模型推理通常存在较高延迟，传统的 HTTP Request-Response 模式会导致前端长时间白屏阻塞，因此 AI 数据流采用异步流式分块传输机制：
* **意图分发**：用户发送文字或图片后，后端 AI 引擎层首先进行 Agent 路由判断（判定由营养专家、健身专家还是综合心理专家接管）。
* **长期记忆检索 (RAG)**：引擎层向底层的本地特征库发起检索，提取该用户的体脂基准、饮食偏好、历史病史等记忆特征。
* **提示词组装**：将检索到的背景上下文与当前用户的输入（如当前的热量缺口）进行动态合并，生成完整的 Prompt 提交给大模型。
* **SSE 流式输出**：Node.js 后端在调用 AI 接口时开启 `stream: true` 模式。无需等待模型全部生成完毕，接收到首个 Token 即刻通过 Server-Sent Events (SSE) 长连接向前端持续推送数据块。前端渲染出“打字机”效果，极大降低用户的感知延迟。

**缓存与持久化数据流 (Cache & Persistence Flow)**
为保护底层 SQLite 数据库免受高频零碎数据（如连续打卡状态更新）的并发 I/O 冲击，系统引入旁路缓存（Cache-Aside）与穿透写入（Write-Through）的混合策略：
* **读高频流**：应用层发起读取请求优先查询内存缓存池，若命中直接返回；未命中则穿透查询 SQLite 主库，将结果回写缓存后返回前端。
* **写聚合流**：非核心时序健康日志先快速写入缓存池，由后端定时任务批量持久化到 SQLite；核心配置数据（如修改密码、重置目标）则采用同步机制直接穿透落盘，确保强一致性。

### 3.3 具体 ui 设计
#### 3.3.1 登录与注册视图 (Auth View)
结构：全屏居中布局。主视觉为一个包含毛玻璃效果（Glassmorphism）的白板卡片。

组件映射：

顶部使用 Radix Tabs 组件实现“登录/注册”无刷新切换。

密码框包含内置的 SVG 眼睛图标，点击切换 type="password" 与 type="text"。

错误反馈：使用 React Hook Form 捕获错误，触发输入框 Tailwind 类名切换（如 border-red-500），并在底部弹出全局 Toast。

#### 3.3.2 健康主工作台 (Dashboard View)
结构：移动端优先的流式垂直布局。

组件映射：

头部区：展示用户头像与当日问候语。

核心指标卡（Cards）：利用 SVG 绘制的环形进度条（Progress Ring），中心显示大字号的剩余卡路里数值。

快捷操作栏（FAB - Floating Action Button）：固定在屏幕底部的悬浮按钮组，包含核心的“拍照（Snap）”和“手动记录”主功能入口，点击具有轻微的缩放反馈（active:scale-95）。

#### 3.3.3 拍照识图视图 (Snap View)
结构：沉浸式多模态交互界面。

组件映射：

相机图层：全屏覆盖的视频流容器，中心带有半透明的对焦框提示。

加载反馈态（Skeleton & Loader）：在等待后端多模态 API 响应时，界面覆盖骨架屏，并伴有自上而下的扫描线动画（通过 Tailwind animate-pulse 及自定义 keyframes 实现）。

结果抽屉（Bottom Sheet）：识别成功后，使用 Radix Dialog 模拟从底部弹出的抽屉面板，以列表形态展示识别到的食物项，支持左滑删除或点击修改克数。

#### 3.3.4 AI 专家对话室 (Agent Chat View)
结构：典型的 IM（即时通讯）全屏气泡布局，底部固定输入框。

组件映射：

专家切换栏：页面顶部的横向滚动胶囊按钮（Pills），切换不同的 AI 角色（如营养师、教练）。

消息气泡：根据数据源区分样式。用户气泡靠右对齐（品牌色底色），AI 气泡靠左对齐（浅灰/暗色底色）。

流式打字机交互：接收到 SSE 数据流时，通过 React 状态逐字更新文本内容。末尾动态渲染一个闪烁的光标（animate-blink），增强“AI 正在思考和打字”的陪伴感。

# 5 性能指标设计
## 5.1 响应时间指标
1. 通用接口请求整体响应时间 ≤ 2s；健康日志提交、本地数据同步操作响应 ≤ 1s。
2. 卡路里计算、AI健康建议生成、长期数据趋势分析报告加载响应时间 ≤ 3s。
3. 对话交互、记忆读取与上下文加载响应时间 ≤ 1.5s，保障多轮对话流畅性。

## 5.2 数据处理能力指标
1. 单用户单日支持50条及以上饮食、运动、状态日志记录的存储、解析与检索。
2. 累计1年历史健康数据整合分析、可视化图表渲染耗时 ≤ 5s。
3. 适配学生项目使用场景，支持同时在线用户数≥200，系统无明显卡顿、页面无阻塞。
4. 内置食物数据库、用户记忆数据轻量化加载，减少重复计算资源消耗。

## 5.3 图片处理性能指标
1. 食物拍摄图片压缩、解析、AI识别全流程处理时间 ≤ 5s，支持1080P高清图片无损压缩。
2. 数据分析截图、报告导出截图生成耗时 ≤ 2s，满足日常截图留存需求。
3. 支持单张图片轻量化预处理，降低前端与后端运算压力。

## 5.4 系统稳定与资源指标
1. 系统持续运行稳定性良好，无频繁崩溃、卡死、内存溢出问题。
2. 优化内存调度机制，规避内存泄漏问题，长时间后台运行资源占用可控。
3. 本地缓存、数据读写高效稳定，保障离线基础功能正常使用。

## 7.2 AI Memory系统设计
### 7.2.1 整体架构设计
采用多专家独立记忆架构，划分营养师、健身教练、健康顾问独立记忆空间，同时搭建公共共享记忆层与上下文管理层，实现记忆隔离、同步与调度，规避不同角色信息混淆干扰。

### 7.2.2 记忆管理设计
1. 记忆持久化：所有AI角色记忆数据本地持久化存储，支持长期保存，重启系统不丢失数据。
2. 记忆版本与内容管理：限制记忆文本长度与格式规范，轻量化存储，同时保留记忆变更基础记录，实现简易版本回溯。
3. 独立记忆空间：为不同AI专家角色分配隔离记忆存储空间，保证各角色专业建议的独立性。
4. 记忆共享与清理：支持多角色间基础用户数据共享，实现综合健康分析；提供手动清理指定角色记忆、批量清空记忆功能，兼顾隐私管理。

### 7.2.3 上下文管理设计
1. 标准化上下文协议适配，维持对话上下文连续性，实现跨会话长期记忆留存。
2. 建立记忆实时更新机制，用户健康数据、对话内容变更后，自动同步更新至AI记忆库。
3. 优化轻量化检索机制，通过用户标识、角色标识快速检索对应记忆内容，保障对话响应效率。
4. 支持多角色对话上下文快速切换，无缝切换营养师、健身顾问对话场景。

### 7.2.4 接口与安全设计
1. 提供标准化记忆增删改查接口，支撑记忆读取、更新、同步、备份全流程操作。
2. 增加记忆数据格式校验、内容验证机制，过滤无效数据，保证记忆内容规范有效。
3. 完善记忆访问权限控制，仅授权AI角色可读取对应记忆内容，敏感记忆加密存储。
4. 配套记忆数据备份与恢复功能，支持手动导出记忆数据，防止个性化记忆内容丢失。

### 7.2.5 联动交互设计
1. 智能对话系统与AI记忆深度联动，多轮对话内容自动同步入库，持续完善用户健康画像。
2. AI依托历史记忆、7天健康数据、多维度生活状态信息，生成贴合用户实际的个性化建议。
3. 依托动态记忆能力，记录用户饮食偏好、运动习惯、健康目标，持续优化AI建议精准度。
