# 1. 项目立项及描述
## 1.1 项目名称
菲特勒斯—你的个人健康管理专家
## 1.2 项目描述
### 1.2.1 项目背景
当今，数字化健康管理领域处于关键转型期。随着人们生活水平提升和对健康生活追求增强，传统健康管理工具局限凸显，多停留在简单数据记录，缺乏多维数据整合分析能力，健康建议无个性化，用户体验差、粘性低，难以改善健康。但随着健康意识和可支配收入增加，尤其在后疫情时代，消费者对智能化、个性化健康管理服务需求攀升，带来新机遇与挑战。“菲特勒斯 —— 你的个人健康管理专家” 项目顺势而生，将运用多模态学习、动态记忆网络等技术，构建理解用户全方位健康状态的智能系统，实现个性化健康管理。
### 1.2.2 客户与最终用户
**客户**涵盖健康科技企业、医疗机构和保险公司。
*   **健康科技企业**：期望引入 “菲特勒斯” 先进智能健康管理系统，丰富产品线，提升竞争力，获取新增长点。
*   **医疗机构**：可将其作为辅助工具用于患者健康管理与疾病预防，借助智能决策和个性化建议提高医疗服务质量。
**最终用户**包括追求健康生活方式的人群、患有慢性疾病的人群以及关注心理健康的人群。
*   **追求健康生活者**：可借助系统规划饮食、运动和生活习惯以实现健康目标。
*   **慢性病患者**：能实时掌握健康状况，获个性化治疗与指导，提升生活质量。
### 1.2.3 产品范围
“菲特勒斯—你的个人健康管理专家”项目旨在构建下一代智能健康管理平台，内容包括：
*   **功能模块**：
    *   智能健康数据采集系统：用计算机视觉等采集多源数据。
    *   智能决策支持系统：用多专家 AI 协同为用户提供个性化建议与实时方案调整。
    *   用户交互系统：界面简洁易用，支持用户互动。
*   **技术架构**：构建稳定高效、具备可靠性与可扩展性的架构，采用云、大数据、人工智能技术保障数据安全。制定详细实施计划，明确各阶段任务与时间节点；建立质量保障体系严格测试评估；制定长期运营维护计划，及时响应用户反馈并优化系统。

# 2. 需求分析
## 2.1 运行环境需求
### 硬件环境
*   **移动端设备**：
    *   智能手机：支持 iOS 14.0 及以上版本、Android 10.0 及以上版本；配备 $\ge$ 800 万像素且支持自动对焦的后置摄像头；处理器性能 $\ge$ 骁龙 660/天玑 700 /苹果 A11 及同等水平；内存 $\ge$ 4GB；存储空间 $\ge$ 16GB。
    *   平板设备：支持 iOS 14.0/Android 10.0 及以上系统；屏幕尺寸 $\ge$ 7 英寸。
*   **桌面端设备（可选）**：
    *   支持 Windows 10 及以上、macOS 11 及以上系统；处理器 $\ge$ Intel i3/AMD Ryzen 3；内存 $\ge$ 8GB；存储空间 $\ge$ 50GB；适配网页端或客户端数据同步与可视化展示。
### 软件环境
*   **操作系统兼容性**：
    *   移动端：深度适配 iOS、Android 主流系统版本并支持系统自动更新后功能兼容。
    *   桌面端：支持 Chrome 90.0+、Edge 90.0+、Safari 14.0+ 等现代浏览器或独立客户端适配 Windows/macOS 系统。
*   **依赖服务**：
    *   需启用设备相机权限（饮食拍照）、存储权限（本地保存建议、报告、截图）、网络权限（数据同步与 AI 分析）。
    *   集成时间同步服务（确保日志时间准确性）、通知服务（提醒记录健康数据）。
## 2.2 系统角色
### 1) 普通用户
核心使用角色，通过移动端或桌面端设备使用产品核心功能。
*   **主要职责**：
    *   记录每日体重、消耗、饮食、运动数据和每日状态等。
    *   分析健康数据（查看净卡路里计算、健康趋势分析报告）。
    *   与 AI 进行智能对话，解答各类疑惑。
    *   接收并提出个性化建议，制定健康计划。
    *   完善并且可以导出个人资料和 AI 建议。
### 2) 系统管理员
负责产品后台运维与管理。
*   **主要职责**：
    *   维护食物库、运动项目库等基础数据。
    *   监控系统运行状态，处理用户权限配置、数据备份与恢复。
    *   优化数据展示规则和系统兼容性。
### 3) AI 专家
产品内置的虚拟角色集群，包括营养顾问、运动教练、心理顾问等细分角色。
*   **主要职责**：
    *   基于用户健康数据和深度档案，共享数据记忆并提供多视角建议。
    *   支持会诊模式下的协同分析。
## 2.3 功能需求
### 2.3.1 功能性需求概括
该产品功能性需求涵盖七大核心模块。
1.  **健康日志记录**：支持多模态录入。
2.  **数据系统分析**：能依据饮食、运动及个人数据计算净卡路里，对长期数据进行趋势分析并生成可视化报告。数据展示会在有效数据点少时自动调整范围，至少显示 5 天数据并支持多种形式。
3.  **个性化 AI 私教**：基于深度健康档案提供精准建议。
4.  **智能建议**：根据数据和专家角色生成饮食、运动等个性化建议且可本地保存。
5.  **食物热效应分析**：能识别影响因素并给出增强建议，报告可保存分享。
6.  **其他功能**：支持健康数据页面完整截图，系统自动优化样式提供高分辨率紧凑截图。

## 2.4 非功能需求
### 2.4.1 性能需求
1.  **响应速度**：
    *   饮食拍照识别、自然语言解析食物/运动信息：$\le$ 2 秒。
    *   健康日志提交、数据同步操作：$\le$ 1 秒。
    *   净卡路里计算、AI 建议生成及趋势分析报告加载：$\le$ 3 秒。
2.  **数据处理能力**：
    *   支持单用户日均 50+ 条日志记录（含文字、图片）的高效存储与检索。
    *   长期数据（$\ge$ 1年）分析时可视化报告生成时间 $\le$ 5 秒。
    *   同时在线用户数 $\ge$ 1000 时，系统无明显卡顿或延迟。
3.  **兼容性性能**：
    *   在适配的硬件设备上，数据展示、页面截图功能加载流畅。
    *   截图生成时间 $\le$ 2 秒，且支持高分辨率（$\ge$ 1080P）输出无失真。
### 2.4.2 安全性需求
1.  **数据加密**：
    *   用户健康日志、深度档案本地存储采用 AES-256 加密。
    *   数据传输全程使用 HTTPS 协议，云端存储数据需加密备份，防止未授权访问。
2.  **权限管控**：
    *   严格限制系统权限，相机、存储、网络等敏感权限仅在对应功能触发时临时授权。
    *   用户数据仅对本人及授权的 AI 专家角色开放，系统管理员无权限查看具体个人健康内容。
3.  **隐私保护**：
    *   符合个人信息保护法规，明确告知用户数据收集范围与用途。
    *   支持数据本地删除、账号注销时的全量数据清除功能。
    *   禁止未经用户允许的数据分析结果共享或商业化使用。
4.  **防篡改与备份**：
    *   健康数据记录生成后自动添加时间戳与校验机制，防止恶意篡改。
    *   系统每日自动备份用户数据，支持 30 天内的数据回溯与恢复。

# 3. 设计思想
## 3.1 系统架构设计
### 3.1.1 总体架构
本系统采用前后端分离的架构，前端使用 React 构建用户界面，后端使用 Java 语言和 MySQL 数据库进行数据处理和存储。这种架构的优点是前后端可以独立开发、部署和维护，提高了开发效率和系统的可扩展性。
### 3.1.2 分层架构
*   **表示层**：负责与用户进行交互，接收用户的输入并展示系统的输出。在本系统中，前端的 React 应用属于表示层。
*   **业务逻辑层**：处理系统的核心业务逻辑，如用户认证、健康数据处理、专家对话等。后端的 Java 服务属于业务逻辑层。
*   **数据访问层**：与数据库进行交互，实现数据的增删改查操作。后端的 DAO 类属于数据访问层。
*   **数据层**：存储系统所有数据，如用户信息、专家对话记录等。MySQL 数据库属于数据层。
## 3.2 系统模块设计
### 1) 前端模块
*   **① 用户界面模块**：负责构建系统的用户界面，包括登录、注册页面、健康数据展示页面等。
*   **② 交互逻辑模块**：处理用户的交互事件，如按钮点击、表单提交等，并与后端进行数据交互。
*   **③ 状态管理模块**：管理系统的状态，如用户信息、健康数据等，确保数的一致性和可维护性。
### 2) 后端模块
*   **① 用户管理模块**：负责用户的注册、登录、信息修改等操作，确保用户的身份安全。
*   **② 健康数据管理模块**：处理用户的健康数据，如体重记录、食物摄入记录、运动记录等，并提供数据的查询和统计功能。
*   **③ 专家对话模块**：实现用户与专家的对话功能，包括消息的发送、接收、存储和查询等。
*   **④ 智能建议模块**：根据用户的健康数据和历史对话记录，为用户提供个性化的健康建议，并记录建议的历史和效果。
*   **⑤ 数据库管理模块**：负责数据库的初始化、状态检查、数据清理和备份等操作，确保数据库的正常运行。

### 3.3.2 数据库关系
*   **一对多关系**：用户表（`user_profiles`）与体重记录表（`weight_records`）、活动水平记录表（`activity_level_records`）、食物记录表（`food_entries`）、运动记录表（`exercise_entries`）、专家对话表（`chat_{expert_role}`）、建议历史表（`suggestion_history`）之间存在一对多的关系，通过 `user_id` 字段进行关联。
*   **表结构一致性**：专家对话表（`chat_{expert_role}`）之间的表结构相同，只是表名不同，分别对应不同的专家角色。
