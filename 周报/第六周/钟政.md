### 🧠 突破上下文窗口的物理囚笼
在Snapifit-AI的众多设计中，我最兴奋、也最头疼的就是7.2节的【AI Memory 系统设计】。为什么？因为大模型（LLM）的上下文窗口是有限的且昂贵的。如果用户用了半年，我们不可能每次对话都把半年的数据塞给OpenAI。

**我本周的核心使命，就是设计一套“体外记忆”系统。**
这不是简单的缓存，而是一套**基于角色隔离的持久化知识图谱简化版**。

### 🧬 记忆数据模型的进化论
在设计 7.2.2 的 `AIMemory` 接口时，我推翻了三版方案：

*   *V1.0 (废弃)*：把所有聊天记录存起来。👉 问题：噪音太大，检索效率低。
*   *V2.0 (废弃)*：让AI自己总结一段长文本存起来。👉 问题：容易丢失细节（比如忘了用户不吃香菜）。
*   *V3.0 (最终版)*：采用结构化 + 半结构化混合。

最终呈现的核心数据结构如下：
    expertId: string;   // 绝对核心：营养师和健身教练的记忆必须隔离！
    content: string;    // 存储经过AI提炼的结构化标签，如 "[allergy: peanut], [habit: stay_up_late]"
    lastUpdated: string;
    version: number;    // 为什么需要版本？防止多标签页并发对话时，旧的记忆覆盖新的记忆（乐观锁机制）

### 🗄️ 为什么是 IndexedDB 而不是云端？
7.2.3节明确了存储机制为本地数据库的 `aiMemories` 对象。
有人可能会问：既然后端有MySQL，为什么记忆放浏览器的IndexedDB？
**这是我深思熟虑的架构决策：隐私与速度。**
1.  **隐私**：记忆里包含了最敏感的健康隐私（疾病史、心理状态）。放在本地，意味着我们（服务端）触碰不到这些明文数据，从根本上规避了合规风险（呼应了王增益的安全合规设计）。
2.  **速度**：每次对话前加载记忆，读本地是毫秒级，读网络是百毫秒级，这直接保住了郭栋定下的“文本解析≤2s”的指标。

### 🛡️ 记忆的安全与湮灭
在 7.2.5 错误处理中，我设想了一个极端场景：**用户开启浏览器无痕模式，或者清空了缓存**。
这时候IndexedDB会写入失败。我的兜底策略是：降级为会话级存储（仅当次会话有效），并在UI上静默提示“当前为隐私模式，AI记忆将不会保留”。

而 7.2.7 的加密设计，我强制要求内容字段在落盘前必须经过前端 AES 加密，即使黑客拿到了用户的电脑，也读不懂他的健康档案。
