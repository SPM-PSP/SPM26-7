# Snapifit-AI 项目组第六周工作总结报告

**项目名称**：Snapifit-AI 综合性健康管理与AI辅助系统  
**报告周期**：2026年4月中旬（第六周）  
**当前阶段**：系统概要设计（SDD）阶段  
**核心产出**：《Snapifit-AI 设计文档 (SDD) v1.0》

---

## 一、 本周工作概述
本周项目组正式由需求分析阶段全面跨入系统概要设计（SDD）阶段。团队以“架构解耦、性能量化、安全兜底、AI赋能”为核心指导思想，完成了系统七大核心章节的设计工作。从顶层的四层架构搭建，到中观的功能模块拆解与接口契约制定，再到微观的AI记忆模型与硬件资源规划，团队成功将抽象的业务诉求转化为了具备强工程约束力的技术蓝图，为后续的详细设计与编码实现奠定了坚实基础。

## 二、 核心设计产出与决策

### 1. 架构演进与技术选型定调
系统确立了“前后端分离+分层架构”的稳健底座，选定 `React 18 + Java 11+ + MySQL 8.0` 作为核心技术栈。在架构模式上采取了务实的混合策略：以经典分层架构满足MVP快速迭代，同时在设计规范中预留微服务拆分与服务发现机制，并明确了事件驱动与CQRS模式的演进路径，兼顾了当前的开发效率与未来的水平扩展需求。

### 2. 功能矩阵精细化与接口契约化
完成了五大核心模块（前端交互、后端服务、数据层、AI服务、系统设置）的颗粒度拆解，建立了业务域到技术实现层的清晰映射。在此基础上，全盘梳理了10大接口组、数十个RESTful端点，彻底贯彻资源导向设计原则。针对AI流式输出与图片识别等非常规场景，创新性地引入了异步状态轮询与记忆同步机制，实现了前后端交互契约的无歧义对接。

### 3. AI Memory 体外记忆架构创新
突破了大模型上下文窗口的限制，自主设计了一套基于 `IndexedDB` 的多专家角色持久化记忆系统。通过 `expertId` 实现角色隔离，采用结构化标签提取代替全量对话存储，并引入 `version` 字段解决多端并发冲突。该设计将核心隐私数据下沉至本地，在保障毫秒级加载性能的同时，从物理层面实现了健康数据的隐私合规隔离。

### 4. 性能指标量化与硬件资源对齐
摒弃了模糊的体验描述，构建了涵盖接口响应、AI推理、存储检索、并发承载的“五维性能基准账本”（如：P95 ≤1s，QPS ≥300，1年聚合 ≤2s）。基于该性能基线，进行了精准的异构硬件资源规划——区分了接入层的高可用冗余、应用层的计算配额以及数据库层的NVMe高速IOPS需求，确保硬件投资直接转化为系统性能指标。

### 5. 纵深防御与韧性系统设计
构建了从外网WAF到内网审计的全链路安全防线，并在系统质量属性中重点强化了“依赖隔离与降级”策略。明确规定了当第三方AI服务不可用时的本地规则引擎兜底方案，配合蓝绿部署与全链路追踪设计，保障了系统在极端场景下的高可用性（≥99.9%）与快速自愈能力。

## 三、 遗留风险与改进方向
1. **前端状态管理瓶颈**：当前选型的 `React Context API` 在AI多轮对话的高频状态流转场景下，可能面临不必要的全局重渲染性能损耗，需在下周启动针对性PoC验证。
2. **AI记忆检索策略未闭环**：目前已解决记忆的“存”与“隔离”，但针对海量本地记忆的“精准提取（如向量检索降级方案）”及“记忆衰减机制”仍需进一步设计。
3. **硬件方案的云原生适配**：当前硬件规划偏向传统物理机视角，需尽快转化为符合K8s容器化部署的逻辑资源配额与云厂商具体采购清单。

## 四、 下周工作规划
1. **研发环境搭建**：基于SDD架构，前后端完成本地开发环境配置与基础工程初始化。
2. **接口Mock先行**：利用Apifox等工具将接口文档转化为可用的Mock服务，解除前端页面开发的后端依赖。
3. **压测脚手架准备**：搭建JMeter基础压测环境，针对核心查询与写入接口进行基线摸底测试。
4. **AI模块深度验证**：开展“体外记忆”注入Prompt的端到端Demo测试，量化评估AI回复准确率的提升幅度。
