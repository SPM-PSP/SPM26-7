### 🎯 本周核心命题定调
当项目从纯粹的业务诉求（SRS）向工程落地（SDD）跨越时，第一脚必须踩实。本周我的主要精力没有放在具体的代码逻辑上，而是**悬浮在系统之上做“减法”和“切分”**。负责统稿《Snapifit-AI设计文档》的第一、第二章，也就是为整个系统定调子、立骨架。

过去我们常说“这是一个健康管理APP”，但在1.1和1.2节的梳理中，我将系统定位强行拉升到了四个维度：
*   **健康管理平台**（基础底盘）
*   **AI辅助系统**（技术护城河）
*   **用户交互中心**（体验抓手）
*   **数据驱动平台**（长期价值）

这不仅仅是一句口号，它直接决定了后续2.1节整体架构图不能是单体，必须是分层解耦的。

### 🏗️ 架构模式的取舍博弈
在2.4节架构模式的详细设计中，我其实经历了不小的纠结。最终呈现在文档里的是一套**“分层架构为基座，微服务与事件驱动为演进方向”**的混合策略。

    [演进路径] 
    单体分层 (MVP阶段) ➔ 核心微服务拆分 (增长期) ➔ 事件驱动/CQRS (成熟期)

为什么不直接上微服务？因为对于当前团队规模，过早引入服务发现、Saga分布式事务（详见2.4.3），带来的运维复杂度会拖垮开发速度。因此，我在2.2节技术选型中，坚定地锁定了 `React 18 + Java 11+ + MySQL 8.0` 这个成熟稳健的组合，而在2.9节预留了水平扩展的插件化接口。

### 🗄️ 数据层的心智模型转换
除了架构，本周另一个重头戏是2.11节的数据库设计规划。传统做法是上来就画E-R图，但我要求先明确数据关系模式。

针对我们的核心业务，我抽象出了两大类数据关系：
1.  **时序型高频数据**：如 `food_entries`（饮食）、`exercise_entries`（运动），这类数据特点是“只追加、少修改、按时间范围查询”，后期必须走向分区甚至归档。
2.  **状态型低频数据**：如 `user_profiles`，强依赖事务一致性。

基于此，我敲定了 `MySQL主从复制 + 读写分离` 的底层数据架构（2.11.1），为后续郭栋同学做性能压测提供了物理基础。

### 📝 个人复盘
文档的顶层设计最容易陷入“空对空”的陷阱。本周在撰写2.5（交互设计）、2.6（错误处理）时，我尽量逼迫自己把抽象原则对应到具体的业务场景中。比如错误分类，不再是简单的“报错”，而是明确拆分为业务错误（如卡路里计算溢出）、系统错误（如DB超时）、安全错误（如SQL注入），这样才对下游开发有指导意义。
